# Front-end 보안 점검 체크리스트

React/TS SPA에 특화된 항목이다. 각 항목을 변경사항에 대입한다.

## 1. XSS / 렌더 안전성

- [ ] `dangerouslySetInnerHTML`을 외부/사용자 입력에 사용하지 않는가? 불가피하면 sanitize(DOMPurify 등)를 거치는가?
- [ ] `innerHTML`, `document.write`, `eval`, `new Function`을 사용하지 않는가?
- [ ] URL을 렌더/이동에 쓸 때 `javascript:` 스킴을 차단하는가? (`href`, `src`, 리다이렉트)
- [ ] 마크다운/리치텍스트 렌더러에 raw HTML 허용 옵션이 켜져 있지 않은가?

## 2. 시크릿 / 환경변수

- [ ] API key, 토큰, 비밀번호가 소스에 하드코딩돼 있지 않은가?
- [ ] `import.meta.env` / `process.env`로 번들에 들어가는 값 중 브라우저에 노출되면 안 되는 것이 없는가?
- [ ] `.env`가 커밋되지 않고 `.gitignore`에 있는가? (`.env.example`만 커밋)
- [ ] 서버 전용 시크릿(Gemini API key 등)을 클라이언트에서 직접 호출하지 않는가? (프록시 필요 여부 확인)

## 3. 브라우저 저장소 / 인증

- [ ] localStorage/sessionStorage/cookie에 토큰·개인정보 등 민감정보를 저장하지 않는가?
- [ ] 인증 토큰을 저장한다면 XSS 노출·만료·초기화 정책이 있는가?
- [ ] 쿠키 사용 시 `Secure`, `HttpOnly`(서버 설정), `SameSite`를 고려했는가?

## 4. 네트워크 / API

- [ ] 엔드포인트가 `https`인가? 평문 `http://`가 없는가?
- [ ] 에러 응답의 raw `message`/stack/내부 endpoint를 사용자에게 그대로 노출하지 않는가? (`code` 기반 매핑 사용)
- [ ] 사용자 입력이 쿼리스트링/경로에 들어갈 때 인코딩·검증하는가?
- [ ] `fetch`에 불필요하게 `credentials: "include"`를 넣지 않았는가?

## 5. 외부 링크 / 리소스

- [ ] `target="_blank"` 링크에 `rel="noopener noreferrer"`가 있는가?
- [ ] 외부에서 받은 URL로 이동/렌더할 때 신뢰 도메인을 검증하는가?
- [ ] 서드파티 스크립트/iframe을 무분별하게 삽입하지 않는가?

## 6. 입력 / 폼

- [ ] 사용자 입력을 신뢰하지 않고, 클라이언트 검증은 UX용일 뿐 서버 검증을 대체하지 않음을 전제하는가?
- [ ] 파일 업로드 시 타입/크기 제한을 확인하는가?
- [ ] 정규식에 ReDoS 위험(중첩 수량자)이 없는가?

## 7. 의존성 / 빌드

- [ ] 새로 추가한 패키지의 신뢰성·필요성을 확인했는가?
- [ ] `pnpm audit` 수준의 알려진 취약점을 확인했는가?
- [ ] source map이 프로덕션에 시크릿을 노출하지 않는가?

## 8. 정보 노출

- [ ] `console.log`에 토큰·개인정보·응답 전체를 남기지 않는가?
- [ ] 에러 바운더리/화면에 스택트레이스를 그대로 보여주지 않는가?

## 심각도 기준

- **Critical**: 시크릿 노출, 저장형 XSS, 인증 우회.
- **High**: 반사형 XSS, 민감정보 저장, 평문 전송.
- **Medium**: `rel` 누락, 과도한 로깅, 약한 입력 검증.
- **Low**: 방어적 개선 권고.
